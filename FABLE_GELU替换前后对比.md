# FABLE 替换 SIGMA GELU：替换前后对比

统一口径：BERT-Tiny 单个 Transformer Block，sequence length 128，FFN hidden size 512，共 65,536 次 GELU 查询。

## 核心结论

在只有 256 项的 GELU 修正表场景中，FABLE 没有带来性能收益：协议时间约慢 **728.6 倍**，通信量约增加 **11,155 倍**。

主要原因是 FABLE BatchPIR 至少按 65,536 项数据库处理，导致 256 项 GELU 表被填充扩大 256 倍，并且 65,536 次查询需要拆成 16 批执行。因此，FABLE 更适合 XLM-R Embedding 这类大规模查表，不适合替换小型 GELU LUT。

## 替换前后对照

| 指标 | 替换前：SIGMA GELU | 替换后：FABLE GELU | 变化 | 关键意义 |
|---|---:|---:|---:|---|
| Hidden size | 128 | 128 | 相同 | 控制模型规模变量 |
| Attention heads | 2 | 2 | 相同 | 模型配置一致 |
| FFN hidden size | 512 | 512 | 相同 | 决定每个 token 的 GELU 数量 |
| Sequence length | 128 | 128 | 相同 | 两边工作负载一致 |
| 查询数 / Block | 65,536 | 65,536 | 相同 | 进行每 Block 对照的基础 |
| 逻辑 LUT 项数 | 256 | 256 | 相同 | 两者计算同一个 GELU 修正函数 |
| 协议实际 LUT 项数 | 256 | 65,536 | **256×** | FABLE 小表填充是开销膨胀的主要来源 |
| 查询批次数 | SIGMA 内部协议 | 16 × 4,096 | 16 批 | FABLE 需要重复执行批处理协议 |
| 在线/协议时间 / Block | 103.539 ms | 75,437 ms | **慢 728.6×** | 决定在线推理延迟，越低越好 |
| 实际 Wall time | 未单独记录 | 85,000 ms | 约 85 秒 | 包含启动、同步等额外时间 |
| 通信量 / Block | 1,736,704 B | 19,372,844,388 B | **高 11,155×** | 决定跨机器带宽和通信成本 |

| 通信量 / query | 26.5 B | 约 295,606 B | **高 11,155×** | 排除查询数量影响后的单次查询成本 |
| 正确性 | 零输入 SIGMA smoke 通过 | 16 chunks 全部零错误 | 均通过各自检查 | 说明协议可运行，但两种测试输入并不完全等价 |

## 指标的关键意义

### 在线/协议时间

表示用户发起推理后需要等待的核心协议时间。该指标越低越好。本实验中，FABLE 将单 Block 的 GELU 时间从约 0.104 秒增加到约 75.4 秒。

### 通信量

表示两方协议需要传输的数据量，直接影响跨服务器带宽、网络费用和广域网延迟。FABLE 每 Block 需要约 18.0 GiB，而 SIGMA 原始 GELU 约为 1.66 MiB。

### 协议实际 LUT 项数

逻辑 GELU 表只有 256 项，但 FABLE BatchPIR 的最小参数要求使其必须填充到 65,536 项。这个 256 倍的无效扩张是 FABLE 在该场景表现较差的根本原因。

### 查询吞吐量

表示每秒能够完成多少次 GELU 查询。该指标排除了“总查询数不同”造成的影响，可以直观看到 FABLE 在小表场景中的处理能力下降。

### Bytes/query

表示完成一次查询平均需要多少通信数据。两边查询数相同时，该指标说明通信膨胀来自协议本身，而不是查询工作量不同。

### 正确性

用于确认查表协议没有明显实现错误。FABLE 的 16 个 chunk 全部零错误，但 SIGMA 与 FABLE 当时采用的测试输入不同，因此不能将它表述为相同模型输入上的精度对比。

## 实验边界

- SIGMA 原始数据覆盖两个 Transformer Block，表中已除以 2，归一化为每 Block。
- SIGMA 使用零输入、零权重 smoke test；FABLE 使用随机有效 GELU 索引。
- 当时验证的是独立 FABLE LUT 组件，尚未连接完整 SIGMA 的 live GPU tensor。
- 实验运行在共享服务器上，时间结果应标记为 provisional。
- 该实验能证明 FABLE 不适合 256 项 GELU 小表，但不能推出 FABLE 在所有查表场景中都更慢。

## 推荐汇报表述

> 在 BERT-Tiny 的 GELU 小表场景中，FABLE 替换方案没有取得性能收益。按单个 Transformer Block 归一化后，FABLE 协议时间约增加 728.6 倍，通信量约增加 11,155 倍。其主要原因是 256 项 GELU 表需要填充至 65,536 项才能使用 FABLE BatchPIR。该结果说明 FABLE 不适合小规模 GELU 查表，因此后续研究将 FABLE 用于 XLM-R Embedding 等大规模查表场景。
